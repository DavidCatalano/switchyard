Iteration on Model Runtime Manager Spec

Great start on this spec. It's concise and well-structured. Here are my observations and suggestions organized by priority:

🔴 High Priority Gaps

1. Missing Startup & Bootstrap Behavior

The spec doesn't address how the service initializes. Add a section or expand §11:

MARKDOWN
### 11.1 Startup Sequence
1. Load and validate YAML config
2. Discover Docker socket / container runtime
3. Optionally restore state from disk or inspect running containers
4. Start models with `auto_start: true`
5. Begin listening on API port

2. Container Lifecycle Beyond start/stop

What happens when a container crashes? Or the control plane restarts? §10 needs:

  • Orphan detection: On startup, detect containers that match naming convention but aren't in state
  • Restart policy: Should crashed containers auto-restart?
  • Graceful shutdown: What happens when the control plane exits?

3. Port Allocation is Under-specified

§10.2 says "static or simple incrementing" but doesn't define either approach. Pick one for MVP:

MARKDOWN
### 10.2 Port Strategy (MVP)
- Base port: 8000 (configurable)
- Allocation: sequential from base port
- Tracking: in-memory map of model -> port
- Release: on container stop, port returns to pool
- Conflict: if port in use, skip to next available

4. Configuration is Missing Critical Fields

§6 doesn't specify:

  • Container image (each backend needs a Docker image)
  • Resource limits (even if MVP doesn't do GPU scheduling, memory limits matter)
  • Global settings (Docker network, base port, log level)
  • Environment variable passthrough
YAML
global:
  docker_network: model-runtime
  base_port: 8000
  log_level: info

models:
  qwen-32b:
    backend: vllm
    image: vllm/vllm-openai:latest  # <-- missing
    resources:
      memory: 32g                    # <-- missing
    control:
      ...

🟡 Medium Priority

5. Streaming Response Handling

§9 routing logic doesn't address streaming (a huge part of LLM APIs). Add:

MARKDOWN
### 9.1 Streaming
- Control plane acts as transparent proxy for SSE streams
- Must forward `Transfer-Encoding: chunked` headers
- No buffering of stream content
- Timeout applies to initial connection only, not stream duration

Also resolves one of the open questions in §15.

6. Backend Health Check Integration

§7 defines a health() method on adapters but it's unclear:

  • Is this called proactively or on-demand?
  • What happens when a backend becomes unhealthy while serving requests?

Consider adding a health polling loop to the lifecycle manager.

7. Error Handling Needs More Cases

§12 is too minimal. Add:

  • Config validation error → 500
  • Docker socket unavailable → 503
  • Container failed to start → 500 (with reason)
  • Request timeout → 504
  • Model loaded but backend not healthy → 503

8. Missing API Endpoint: DELETE /models/{model}/config

There's no way to modify configuration after initial load. For MVP, you could either:

  • Accept that config changes require restart (state it explicitly in Non-Goals)
  • Add a hot-reload endpoint

🟢 Nice to Have (MVP)

9. Logging & Observability

Even in MVP, you need:

  • Structured logs for container lifecycle events
  • Request IDs that propagate to backend logs
  • A /health endpoint on the control plane itself

10. API Response Formats

§5 doesn't specify return formats. Add examples:

MARKDOWN
GET /models
Response:
{
  "models": [
    {
      "name": "qwen-32b",
      "backend": "vllm",

      "port": 8001,

    }
  ]
}

11. Container Cleanup Strategy

What happens to stopped containers? Add to §10:

MARKDOWN
### 10.3 Container Cleanup
- Stopped containers are removed immediately (MVP)
- Logs are discarded (MVP)
- Future: configurable retention

🔵 Structural Suggestions

12. Resolve Open Questions in §15

These are blocking decisions for implementation. I'd recommend:

Question Recommendation
Streaming normalization Pass-through for MVP. Normalize later.
Token limit inconsistencies Document in API response; client handles
Adapter vs sidecar for non-OpenAI runtimes Adapter wraps the container. Sidecars add complexity.
Model metadata exposure Add GET /models/{model}/info returning context size, backend type, etc.

13. Consider a "Threat Model" or "Failure Modes" Section

  • Docker daemon dies
  • Config file becomes invalid
  • Two requests try to load same model simultaneously
  • Port exhaustion
